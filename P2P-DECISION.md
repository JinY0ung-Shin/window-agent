# P2P 대안 선택 — 의사결정 문서

## 목적
현재 libp2p 기반 P2P를 제거하고 더 단순한 대안으로 교체한다.
이 문서는 어떤 대안으로 갈지 결정하기 위한 것이다.

## 프로젝트 컨텍스트
- **앱**: window-agent (Tauri + Rust + React)
- **용도**: AI 에이전트 간 메시지 교환 (텍스트 기반)
- **사용 규모**: 소규모 (개인/소그룹, 수십 명 이하)
- **현재 문제**: libp2p가 너무 복잡 (~2000줄, 6개 프로토콜 스택)

## 핵심 요구사항
1. 에이전트 간 메시지 송수신 (텍스트)
2. 초대 기반 연락처 추가
3. 오프라인 시 메시지 대기/재전송
4. 보안 (암호화 + 인증)
5. 유지보수 용이성 (가장 중요한 동기)

## 후보

### A. iroh (n0)
- QUIC 기반 Rust P2P, NodeId로 연결
- NAT 자동 우회, Noise 암호화 내장
- ~500줄로 축소 예상
- 단점: wire protocol 직접 구현, n0 relay 의존, 마이그레이션 복잡

### B. WebSocket 릴레이 서버
- 중앙 서버에 WS 연결, 서버가 메시지 라우팅
- ~400줄(클라이언트) + ~300줄(서버)
- 단점: 서버 운영 필요, E2E 암호화 별도 구현, SPOF

### C. Nostr 프로토콜
- 공개 릴레이에 서명된 이벤트 게시
- ~600줄, 서버 운영 불필요
- 단점: 실시간성 부족, 에이전트 통신에 비최적, 공개 릴레이 안정성

### D. Matrix (matrix-rust-sdk)
- 홈서버 기반 분산 채팅, E2E 암호화 내장
- ~800줄, 오프라인 메시지 자동
- 단점: 무거움, 홈서버 의존, 계정 관리 복잡

### E. HTTP relay + SSE
- REST API + Server-Sent Events
- ~300줄(클라이언트) + ~400줄(서버)
- 단점: 서버 필요, SSE 단방향 제약

## 평가 기준 (가중치)
1. **단순성/유지보수성** (40%) — 교체의 가장 큰 동기
2. **마이그레이션 난이도** (25%) — 현재 코드에서 얼마나 쉽게 전환 가능한지
3. **기능 충족도** (20%) — 요구사항 5가지를 얼마나 잘 충족하는지
4. **운영 부담** (15%) — 서버 운영, 외부 서비스 의존도

## 요청
각 후보를 위 평가 기준으로 점수를 매기고(1-10), 가중치 적용한 총점으로 최종 1개를 추천하라.
추천 시 반드시 해당 프로젝트의 실제 코드(src-tauri/src/p2p/)를 참고하여 현실적인 판단을 해라.
