# P2P Agent Network - Consensus Plan

> Generated by /cplan (Claude + Codex consensus)
> Consensus reached: Round 3
> Date: 2026-03-17

## Requirement

Window Agent 사용자 간 P2P 에이전트 네트워크 — 에이전트가 주체로 다른 사용자의 에이전트와 대화/업무 조율, 사용자는 감독자. 서버 없이 P2P로 연결, 초대 코드로 사용자 식별, 비서+대리인 모드 지원.

## Agreed Plan

### Architecture

- **P2P Layer**: libp2p (Rust) — noise encryption, yamux multiplexing, mDNS (LAN), optional relay
- **Identity**: Ed25519 키쌍 → PeerId, Tauri secure store에 private key 저장
- **Contact Card**: 서명된 연락 카드 (peer public key + multiaddrs + relay hints + expiry + agent metadata)
- **Agent Protocol**: Typed versioned request-response over libp2p stream
- **Security**: Per-contact capability policy, default deny-all, signed handshake with nonce/challenge
- **Mode**: 비서 모드 (Phase 1) → 대리인 모드 (Phase 2)
- **Lifecycle**: Dormant P2PManager at startup, user opt-in activation via `network_enabled` setting

### Key Design Decisions

1. **Dormant-by-default**: P2PManager는 앱 시작 시 dormant 상태로 등록, 사용자가 명시적으로 네트워크를 활성화해야 swarm이 bind/listen
2. **Security-first**: Phase 1에서 모든 원격 피어의 도구/vault/자격증명 접근 hard-block (Rust 레벨)
3. **Isolated summarization**: 비서 모드 LLM 요약은 persona/tools/vault/credentials 없이 격리된 minimal prompt 사용
4. **LLM fallback**: API 키 없거나 LLM 실패 시 raw text/truncated preview로 승인 UI 표시
5. **Idempotent ACK**: 중복 수신 시에도 ACK 재전송하여 무한 재시도 방지
6. **Crash-safe send**: outgoing peer_messages + outbox entry를 네트워크 전송 전에 먼저 persist

## Implementation Steps

### Phase 1: Direct-Dial Trusted Contacts + 비서 모드

#### Step 1. Identity & Key Management `{parallel: false}`
- Actions: Ed25519 키쌍 생성/로드/저장 (Tauri secure store), PeerId 파생, 앱 첫 실행 시 자동 생성
- Files: `src-tauri/src/p2p/mod.rs`, `src-tauri/src/p2p/identity.rs`
- Dependencies: 없음

#### Step 2. Network Service (Manager + Transport + Protocol) `{parallel: false}`
- Actions (merged milestone):
  - a) P2PManager skeleton: managed state container, dormant by default
  - b) Transport: libp2p config (noise + yamux + TCP + mDNS)
  - c) Protocol behaviours: typed request-response behaviour definition
  - d) Swarm wiring: combine transport + behaviours into swarm
  - e) Runtime: background event loop (tokio::spawn), Tauri event emission, reconnect with exponential backoff, graceful shutdown
  - f) Activation: swarm only binds/listens on explicit user opt-in (`p2p_start` command or persisted `network_enabled=true`). `p2p_stop` unbinds and returns to dormant.
- Files: `src-tauri/src/p2p/manager.rs`, `src-tauri/src/p2p/transport.rs`, `src-tauri/src/p2p/protocol.rs`, `src-tauri/src/p2p/envelope.rs`
- Integration: `src-tauri/src/lib.rs` — register dormant P2PManager as Tauri managed state
- Dependencies: Step 1

#### Step 3. Contact Card & Invite System `{parallel: true}`
- Actions: 서명된 Contact Card (peer_pubkey, multiaddrs[], relay_hints[], expiry, agent_name, agent_description), base64 직렬화/역직렬화, card verification
- Files: `src-tauri/src/p2p/contact_card.rs`, `src-tauri/src/p2p/invite.rs`
- Dependencies: Step 1

#### Step 4. Security & Trust Layer `{parallel: true}`
- Actions: First-contact handshake (mutual nonce/challenge exchange → signed response → identity verification), per-contact CapabilitySet (default: messages + agent info only), remote session hard-block on tools/credentials/vault
- Files: `src-tauri/src/p2p/security.rs`, `src-tauri/src/p2p/capability.rs`
- Dependencies: Step 1

#### Step 5. Persistence Layer `{parallel: true}`
- Actions: Rust-driven migration (009):
  - `contacts` (id, peer_id UNIQUE, public_key, display_name, agent_name, agent_description, local_agent_id FK→agents, mode DEFAULT 'secretary', capabilities_json, status, invite_card_raw, created_at, updated_at)
  - `peer_threads` (id, contact_id FK→contacts, local_agent_id FK→agents, title, summary, created_at, updated_at)
  - `peer_messages` (id, thread_id FK→peer_threads, message_id_unique UNIQUE, correlation_id, direction, sender_agent, content, approval_state, delivery_state, retry_count DEFAULT 0, raw_envelope, created_at)
  - `outbox` (id, peer_message_id FK→peer_messages, target_peer_id, attempts DEFAULT 0, next_retry_at, status DEFAULT 'pending', created_at)
- Duplicate handling: INSERT OR IGNORE on message_id_unique, re-send ACK for known messages
- Crash safety: persist outgoing messages + outbox before network send
- Files: `src-tauri/src/db/migrations.rs` (add migration_009), `src-tauri/src/p2p/db.rs`
- Dependencies: Step 1

#### Step 6. 비서 모드 엔진 `{parallel: false}`
- Actions:
  1. Receive typed message → persist to peer_messages (delivery_state=received)
  2. Summarize with ISOLATED LLM call (no persona/vault/tools/credentials, untrusted input)
  3. Fallback: if LLM unavailable, use raw text / truncated preview
  4. Emit `p2p:approval-needed` event with summary
  5. User reviews → writes/edits response → approve
  6. Optional: user can request agent draft (uses local persona, still requires approval)
  7. Send response, update delivery_state
- Files: `src-tauri/src/p2p/secretary.rs`
- Dependencies: Step 2, 4, 5

#### Step 7. Tauri Commands & Events `{parallel: false}`
- Commands:
  - Lifecycle: `p2p_start`, `p2p_stop`, `p2p_status`, `p2p_get_network_enabled`, `p2p_set_network_enabled`
  - Invites: `p2p_generate_invite`, `p2p_accept_invite`
  - Contacts: `p2p_list_contacts`, `p2p_update_contact`, `p2p_remove_contact`, `p2p_bind_agent`
  - Messaging: `p2p_send_message`, `p2p_approve_message`, `p2p_reject_message`, `p2p_request_draft`
  - Threads: `p2p_list_threads`, `p2p_get_thread`, `p2p_get_thread_messages`
- Events:
  - `p2p:connection-state` → `{status: dormant|starting|active|stopping, peer_count}`
  - `p2p:peer-connected` → `{peer_id, contact_name, agent_name}`
  - `p2p:peer-disconnected` → `{peer_id, reason}`
  - `p2p:incoming-message` → `{thread_id, message_id, sender_agent, preview}`
  - `p2p:approval-needed` → `{thread_id, message_id, summary, sender_agent}`
  - `p2p:delivery-update` → `{message_id, state: sending|delivered|failed|queued}`
  - `p2p:error` → `{code, message, peer_id?}`
- Files: `src-tauri/src/commands/p2p_commands.rs`, `src-tauri/src/commands/mod.rs`
- Integration: `src-tauri/src/lib.rs`, `src/services/commands/p2pCommands.ts`
- Dependencies: Step 2-6

#### Step 8. Frontend — Network Settings & Opt-in `{parallel: true}`
- Actions: Network enable/disable toggle in Settings, first-time consent dialog, network status indicator
- Files: `src/components/settings/` (extend SettingsModal), `src/stores/networkStore.ts`
- Dependencies: Step 7

#### Step 9. Frontend — Contacts & Invite UI `{parallel: true}`
- Actions: Contact list with status, generate/accept invite dialogs, contact detail (bound agent, mode, capabilities)
- Files: `src/components/network/ContactList.tsx`, `InviteDialog.tsx`, `ContactDetail.tsx`
- Dependencies: Step 7, 8

#### Step 10. Frontend — Peer Conversation & Approval UI `{parallel: true}`
- Actions: Thread list, conversation view, approval panel (summary + original + editor + draft request + approve/reject), delivery badges, outbox indicator
- Files: `src/components/network/PeerThread.tsx`, `ApprovalPanel.tsx`, `DeliveryBadge.tsx`
- Dependencies: Step 7, 9

## Acceptance Criteria

### Phase 1
- [ ] Ed25519 identity auto-generated/loaded on first launch
- [ ] P2PManager registered as dormant managed state, activates only on user opt-in
- [ ] `network_enabled` setting persisted, network status visible in UI
- [ ] Two LAN instances connect via signed invite code exchange
- [ ] First-contact handshake with nonce/challenge mutual authentication
- [ ] Typed versioned messages sent/received with ACK
- [ ] Duplicate messages handled idempotently (upsert by message_id_unique, re-ACK)
- [ ] Secretary mode: isolated LLM summary → approval UI → user-approved response sent
- [ ] LLM fallback: raw text preview when summarization unavailable
- [ ] All peer messages/threads persisted in SQLite with delivery state
- [ ] Crash-safe send: persist outbox entry before network attempt
- [ ] Local outbox queues unsent messages, resends on reconnect
- [ ] Contact management UI (add/remove/bind agent/view status)
- [ ] Remote peers CANNOT access local tools/vault/credentials (hard block)
- [ ] Graceful shutdown: swarm cleanup, pending messages queued to outbox
- [ ] Reconnect with exponential backoff on connection loss

## Codex Final Assessment

**Strengths:**
- Lifecycle model coherent: dormant manager at startup, explicit user opt-in
- Merged network-service milestone removes implementation-order risk
- Trust model appropriately conservative: signed cards, nonce/challenge, hard Rust-side blocks
- Persistence matches product needs with dedicated tables
- Secretary flow safe with isolated summarization
- Respects codebase conventions (Rust-driven migrations, Tauri command registration, TS facade)

**Overall:** Plan is internally consistent, security-aware, and well aligned with the current Tauri/Rust/Zustand architecture. Approved for Phase 1 implementation.

## Testing Recommendations (from Codex)

Implement a test matrix covering:
- Two-node lifecycle (start, connect, disconnect, reconnect, shutdown)
- Handshake replay prevention (stale nonce rejection)
- Duplicate delivery (ACK-loss → re-ACK, idempotent persist)
- Restart recovery (outbox resend after app restart)
- Hard-block verification (remote peer cannot trigger tool/vault/credential access)
- Dev/test harness: launch two local nodes against same app codepath
