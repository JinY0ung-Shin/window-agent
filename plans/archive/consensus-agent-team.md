# Agent Team — Consensus Plan

> Generated by /cplan (Claude + Codex consensus)
> Consensus reached: Round 6
> Date: 2026-03-19

## Requirement

Agent Team 기능 — 팀장이 구성원에게 업무 위임, 병렬 실행, 사용자 관찰/개입.
v1 Scope: delegate + report 핵심 흐름. ask_agent, @mention, mid-stream targeted intervention → v2.

## v1 Flow

```
User → Leader (LLM) → delegate([A, B], task)
  ├→ Agent A (parallel, auto-tier tools only) → report ─┐
  └→ Agent B (parallel, auto-tier tools only) → report ─┤
                                                         └→ Leader (LLM) → synthesis → User
User intervention: abort_team_run → new message to leader
```

## Architecture Pillars

1. **ExecutionScope**: `{ actor_agent_id, conversation_id, team_id?, team_run_id?, team_task_id?, role: Dm|TeamLeader|TeamMember, trigger: UserInitiated|BackendTriggered }`
2. **ActorExecutionContext (Rust)**: `resolve(ExecutionScope)` → full runtime context. DM + Team unified service.
3. **Multi-Run frontend**: `runsById` Map, per-run approval/abort
4. **Persistent orchestration**: `team_runs` + `team_tasks` + message FK links
5. **First-class cancellation**: `abort_team_run` → fan-out abort via RunRegistry
6. **Backend-only orchestration dispatch**
7. **Orchestration tools auto-injected** in team conversations (not in TOOL_CONFIG)
8. **Tool policy by turn type**:
   - TeamMember (all turns): auto-tier only, no confirm/deny
   - TeamLeader user-initiated turns: full tool set with frontend approval loop
   - TeamLeader backend-triggered turns (synthesis): auto-tier only
   - ActorExecutionContext uses role + trigger to determine tool filtering
9. **Browser tools excluded** from team v1
10. **Prompt assembly**: DM = TypeScript (existing), Team member = Rust (ActorExecutionContext). Gradual migration.

## Agreed Plan

### Implementation Steps

### 1. DB Schema {parallel: false}

- `teams` (id, name, description, leader_agent_id FK, created_at, updated_at)
- `team_members` (id, team_id FK, agent_id FK, role [leader|member], joined_at)
- `team_runs` (id, team_id FK, conversation_id FK, leader_agent_id, status [running|waiting_reports|done|failed|cancelled], started_at, finished_at)
- `team_tasks` (id, run_id FK, agent_id FK, request_id TEXT, task_description, status [queued|running|done|failed|cancelled], parent_message_id, result_summary, started_at, finished_at)
- `conversations`: +team_id TEXT nullable FK
- `messages`: +sender_agent_id, +team_run_id, +team_task_id
- `tool_call_logs`: +agent_id
- Incremental migration via `run_incremental_migrations()`
- **Files**: migrations.rs, models.rs

### 2. ActorExecutionContext (Rust) {parallel: false}

- `ExecutionScope` struct + `Role` enum + `Trigger` enum
- `resolve(scope)` returns: agent record, persona files, system prompt (role-dependent), enabled tools (role+trigger filtered), credential allowlist, workspace path (actor-scoped), memory/vault scope, LLM settings
- Tool filtering:
  - `Dm + any trigger` → existing behavior
  - `TeamMember + any trigger` → auto-tier native only + report orchestration tool
  - `TeamLeader + UserInitiated` → existing tools + delegate + frontend approval
  - `TeamLeader + BackendTriggered` → auto-tier only + delegate
- DM compat: `scope.role = Dm, actor = conversation.agent_id`
- **Files**: actor_context.rs (new), services/mod.rs
- **Dependencies**: Step 1

### 3. Backend: Team CRUD {parallel: true (with Step 2)}

- Models: Team, TeamMember, TeamDetail, TeamRun, TeamTask
- `team_operations.rs`: CRUD + run/task state machine
- `team_commands.rs`: Tauri wrappers
- lib.rs registration
- **Files**: models.rs, team_operations.rs (new), team_commands.rs (new), mod.rs, lib.rs
- **Dependencies**: Step 1

### 4. Conversation/Message API for Team Mode {parallel: true (with Step 3)}

- `create_team_conversation(team_id, leader_agent_id, title?)` → conversation with team_id set
- `get_conversations()`: return team_id. Frontend splits DM (null) vs Team (not null)
- `ConversationDetail`: +team_id exposed
- `save_message()`: +sender_agent_id, +team_run_id, +team_task_id
- `get_messages()`: return new fields per message
- Frontend types + tauriCommands updated
- `conversationStore`: dmConversations[] / teamConversations[] separation
- **Files**: operations.rs, chat_commands.rs, models.rs, types.ts, tauriCommands.ts, conversationStore.ts
- **Dependencies**: Step 1

### 5. Backend: TeamOrchestrator + Cancellation {parallel: false}

- **execute_delegation(conversation_id, run_id, agent_ids, task, context)**:
  - Create team_tasks (queued) per agent
  - resolve(ExecutionScope { role: TeamMember, trigger: BackendTriggered }) per agent
  - tokio::spawn parallel LLM streams; store request_id in team_task
  - Stream events: `team-agent-stream-chunk/done { run_id, task_id, agent_id, ... }`
  - Member tool execution: auto-tier only via ActorExecutionContext
  - Member orchestration tool: report only
- **handle_report(run_id, task_id, summary)**:
  - task → done, save result_summary
  - All tasks done → resolve(ExecutionScope { role: TeamLeader, trigger: BackendTriggered }) → leader synthesis turn (auto-tier only)
- **abort_team_run(run_id)**:
  - For each running task: RunRegistry.abort(request_id)
  - Statuses → cancelled, partial messages → aborted
  - Emit team-run-cancelled event
- **recover_runs()**: startup → running tasks → failed
- Tool dispatcher: orchestration category → TeamOrchestrator
- delegate: leader-only; report: member-only
- **Files**: team_orchestrator.rs (new), tool_commands.rs, api_service.rs
- **Dependencies**: Steps 2, 3, 4

## Risks + Mitigations

- **API cost**: max 3 concurrent agents (configurable)
- **Context growth**: per-agent summary + selective history
- **Streaming**: rAF batch + independent bubbles per agent
- **Crash recovery**: team_runs/tasks persisted + recover_runs() on startup
- **No circular asks** in v1 (ask_agent deferred)
- **No browser** in team v1
- **No confirm-tier** in member/backend-triggered turns v1

## Deferred to v2

- ask_agent (agent-to-agent free-form communication)
- @mention for directing messages to specific agents
- User mid-stream targeted intervention (interrupt specific agent)
- Confirm-tier tool approval protocol for backend-driven turns
- Browser tool support in team mode

## Codex Final Assessment

- ExecutionScope + ActorExecutionContext cleanly captures role and turn-trigger differences
- Split tool policy for TeamLeader resolves approval-path problem practically
- Team conversations have explicit persistence and reload support
- v1 scope is disciplined: delegate + report, auto-tier members, no browser, no confirm backend flows
- Good balance of forward progress with backward DM compatibility
