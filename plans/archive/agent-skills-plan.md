## Requirements Summary

### Problem Statement
현재 TOOLS.md 기반 도구 시스템은 새 기능 추가 시 Rust 코드 변경이 필요하며, 프롬프트 기반으로 에이전트 행동을 확장할 수 없음. Agent Skills 표준을 도입하여 사용자가 SKILL.md 마크다운 파일만으로 에이전트 능력을 확장할 수 있게 함.

### Scope
**In scope:**
- SKILL.md 파싱 및 디스커버리 (Agent Skills 표준 준수)
- Progressive disclosure (3-tier: 카탈로그 → 본문 → 리소스)
- 시스템 프롬프트 주입 (카탈로그 + 활성 스킬 본문)
- 스킬 관리 UI (SkillBar + Agent Editor)
- Per-agent + Global 스킬 지원
- Export/Import 호환
- /특기 슬래시 커맨드

**Out of scope (v2 후보):**
- 모델 기반 자동 활성화 (AI가 스킬 추천/자동 장착)
- 스킬 마켓플레이스 / GitHub 동기화
- 동적 스킬 생성 (에이전트가 런타임에 스킬 생성)

### Assumptions

- 스킬 본문은 대부분 5000 토큰 이하

## Skill Lifecycle

```
1. DISCOVER    → Rust backend scans skills/ dirs, parses YAML frontmatter
2. CATALOG     → Name+description of ALL available skills injected into system prompt
                  (tier-1 progressive disclosure, ~100 tokens per skill)
3. ACTIVATE    → User explicitly toggles skill on (v1: manual only via SkillBar or /특기)
                  Full SKILL.md body loaded and injected into system prompt
4. RESOURCE    → When active skill references files, agent uses existing read_file tool
                  to access scripts/, references/, assets/ within skill directory
5. PERSIST     → Active skill names saved to conversation DB column, restored on reload
```

## Architecture

### Storage Layout
```
app_data_dir/
├── agents/
│   └── {folder_name}/
│       ├── IDENTITY.md, SOUL.md, USER.md, AGENTS.md, TOOLS.md
│       └── skills/                    ← per-agent skills
│           ├── code-review/
│           │   ├── SKILL.md
│           │   ├── scripts/
│           │   └── references/
│           └── summarize/
│               └── SKILL.md
└── skills/                            ← global shared skills
    └── translate/
        └── SKILL.md
```

### Key Design Decisions
1. **Skills ≠ Tools**: Skills inject prompt instructions. Tools execute code. Orthogonal.
2. **Progressive Disclosure (3 tiers)**:
   - Tier 1 (always): Skill catalog (name+description) in system prompt
   - Tier 2 (on activate): Full SKILL.md body injected
   - Tier 3 (on demand): Resource files accessed via existing read_file tool
3. **Backend = Canonical Parser**: Rust parses YAML frontmatter. Frontend consumes typed results only.
4. **Lenient Loading**: Malformed SKILL.md loaded with diagnostics, not rejected. Missing description → "(설명 없음)". Invalid name → directory name.
5. **Per-agent + Global**: Agent skills override global on name conflict.
6. **Token Budget Arbitration**:
   - Priority: Base persona > Summary > Skill Catalog > Active Skill Bodies > Memory Notes
   - Skill bodies: soft cap 2000, hard cap 3000 tokens
   - Over hard cap: activation blocked with UI warning
   - On restore: missing skills skipped with diagnostic, over-budget skills deactivated newest-first
7. **Persistence**: Names only in DB. Body re-loaded from disk on restore (always fresh).

## Implementation Plan

### Step 1: Rust Backend — Skill Discovery, Parsing & Validation
- Create `src-tauri/src/commands/skill_commands.rs`
- Add `serde_yaml` dependency to Cargo.toml
- **SkillMetadata**: `{ name, description, source: "agent"|"global", path, compatibility?, license?, metadata?, diagnostics: Vec<String> }`
- **SkillContent**: `{ metadata: SkillMetadata, body: String, resource_files: Vec<String> }`
- **Commands**: `list_skills`, `read_skill`, `read_skill_resource`
- YAML parsing: split on `---`, serde_yaml, lenient fallback
- Path validation: canonicalize+prefix (same pattern as tool_commands.rs)
- **Dependencies**: None

### Step 2: DB Migration V5 — Conversation Skills State
- `ALTER TABLE conversations ADD COLUMN active_skills TEXT;` (nullable JSON array)
- Operations: `update_conversation_skills`, expose in `get_conversation_detail`
- **Dependencies**: None (parallel with Step 1)

### Step 3: Frontend — Skill Types & Tauri Command Wrappers
- Types in `types.ts`: SkillMetadata, SkillContent
- Wrappers in `tauriCommands.ts`: listSkills, readSkill, readSkillResource, updateConversationSkills
- No frontend YAML parsing
- **Dependencies**: Steps 1-2

### Step 4: Frontend — Skill Store (Zustand)
- `src/stores/skillStore.ts`
- State: availableSkills, activeSkillBodies, activeSkillNames, activeSkillTokens, catalogPrompt
- Actions: loadSkills, activateSkill (token cap check), deactivateSkill, restoreActiveSkills, getSkillsPromptSection
- Duplicate guard, over-budget fallback, missing-skill fallback on restore
- **Dependencies**: Step 3

### Step 5: System Prompt Integration
- Modify `buildConversationContext()`: new `skillsSection` parameter
- Inject `[AVAILABLE SKILLS]` (catalog) + `[ACTIVE SKILLS]` (bodies) sections
- Modify `sendNormalMessage()`, `regenerateStream()`, `selectConversation()`
- **Dependencies**: Step 4

### Step 6: Frontend UI — SkillBar Component
- `src/components/skill/SkillBar.tsx`: horizontal bar above chat (MemoryBar pattern)
- `src/components/skill/SkillChip.tsx`: active/inactive/loading toggle chips
- Token indicator, diagnostics tooltips
- **Dependencies**: Step 5

### Step 7: Agent Editor — Skills Management
- "특기" section in AgentEditor.tsx
- CRUD: create_skill, update_skill, delete_skill Rust commands
- Inline SKILL.md editor
- Global skills read-only (labeled "공유 특기")
- **Dependencies**: Step 3

### Step 8: Export/Import/Sync Compatibility
- **Agent Export (ZIP)**: Include skills/ directory as-is
- **Agent Import (ZIP)**: Extract skills/, lenient validation
- **Conversation Export (JSON)**: Include active_skills field
- **Conversation Import (JSON)**: Restore active_skills, skip missing skills with diagnostic
- **Global skills on import**: Not bundled in agent ZIP (shared separately); missing global skill on import → diagnostic note, not failure
- **Dependencies**: Steps 1-2

### Step 9: Slash Command — /특기
- `/특기` — list available + active
- `/특기 장착 {name}` — activate
- `/특기 해제 {name}` — deactivate
- Local system message (not sent to API)
- **Dependencies**: Step 6

### Step 10: Bootstrap Integration
- Create empty `skills/` directory for new agents
- **Dependencies**: Step 7

### Step 11: Tests
- **Rust 15+**: YAML parsing, lenient fallback, name validation, directory scanning, override precedence, path validation, resource enumeration, CRUD
- **Frontend 20+**: skillStore (load/activate/deactivate/token cap/restore/duplicate), SkillBar/SkillChip rendering, chatHelpers with skills, slash commands, export/import
- **Dependencies**: Steps 1-10

## Technical Considerations

- **Token Budget Arbitration**: Fixed priority ensures skills never crowd out persona or summary
- **Lenient Loading**: Maximizes interoperability with third-party skills
- **Cache**: Metadata cached per agent, body always fresh from disk
- **Security**: Canonicalize+prefix path validation, scoped access
- **Backward Compatibility**: Purely additive — no-skills agents unchanged
- **Resource Access**: Primary path is `read_skill_resource` Rust command (scoped); existing `read_file` tool as fallback for agent-driven access

### Alternative Approaches Considered
1. **Frontend-only parsing**: YAML 파싱을 TypeScript에서 수행 → 거부 (파싱 중복, 보안 우려)
2. **DB에 스킬 본문 저장**: 활성화 시 body를 DB에 캐시 → 거부 (디스크에서 항상 fresh read가 더 단순)
3. **모델 기반 자동 활성화**: AI가 컨텍스트 기반으로 스킬 추천 → v2로 연기 (복잡도 높음)
4. **스킬 슬롯 제한 (RPG 스타일)**: 최대 3개 슬롯 → 거부 (토큰 예산으로 자연스럽게 제한)
