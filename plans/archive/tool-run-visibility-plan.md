## Requirements Summary

### Problem Statement
현재 채팅 UI는 한 번의 도구 실행 묶음도 여러 개의 독립 카드처럼 렌더링한다. `ChatMessage.tsx`는 assistant의 `tool_calls`를 세로 카드 목록으로 보여주고, 이후 `tool` 결과 메시지는 다시 별도 버블로 렌더링한다. 이 구조 때문에 같은 실행 흐름이 여러 조각으로 분리되고, `http_request`나 `browser_*` 같은 연속 호출에서는 화면 높이를 과도하게 소비한다.

### Scope

**In Scope**
- 채팅 렌더 단계에서 도구 실행 묶음을 `tool_run` 단위로 재구성
- 도구 호출과 결과를 하나의 실행 블록 안에서 연결 표시
- 성공 실행의 기본 접힘, 오류/승인 대기 단계의 강조
- 기존 브라우저 스크린샷/스냅샷, `write_file` 미리보기 등 상세 결과 표현 유지
- 기존 대화 기록과 고아(orphan) 도구 메시지에 대한 안전한 fallback 유지
- 단위 테스트와 컴포넌트 테스트 추가

**Out of Scope**
- DB 스키마 변경
- Rust/Tauri 백엔드 계약 변경
- 도구 승인 의미 자체 변경
- 디버그 패널 구조 개편

### Success Criteria
- [ ] 연속된 도구 호출/결과가 다수의 풀사이즈 카드 대신 하나의 grouped execution block으로 렌더링된다.
- [ ] 사용자는 펼치지 않아도 도구 이름, 상태, 요약 결과를 빠르게 스캔할 수 있다.
- [ ] 오류, 거부, 승인 대기 단계는 기본적으로 눈에 띄게 표시된다.
- [ ] 브라우저 스크린샷/스냅샷 등 기존 상세 결과 보기 기능이 유지된다.
- [ ] assistant 텍스트와 `tool_calls`가 함께 있는 메시지도 정상 렌더링된다.
- [ ] 고아 도구 결과나 불완전한 과거 대화도 안전하게 표시된다.
- [ ] 승인 버튼 동작은 현재 의미를 그대로 유지한다.

### Constraints
- 1차 구현은 프론트엔드 렌더링 계층에서만 해결한다.
- 원본 `messages` 배열은 API 컨텍스트 구성과 재생성 흐름을 위해 canonical transcript로 유지한다.
- `toolRunStore`의 현재 상태 모델을 최대한 재사용한다.
- 모바일에서도 줄바꿈과 접힘 상태가 읽기 쉬워야 한다.

### Assumptions
- 현재 아키텍처상 동시에 활성 상태인 live tool run은 하나다.
- 정상 저장된 대화에서는 assistant의 `tool_calls` 뒤에 해당 `tool` 결과가 이어지고, 그다음 assistant 응답이 온다.
- 일부 과거/가져온 대화에는 부모 없는 `tool` 메시지나 파싱 실패가 존재할 수 있다.

## Architecture

### Approach
원본 메시지 저장 구조는 그대로 두고, `ChatWindow.tsx` 렌더 경계에서만 화면용 view model을 만든다. 핵심은 `ChatMessage` 내부에서 억지로 형제를 참조하지 않고, 먼저 `ChatMessage[]`를 `RenderBlock[]`로 변환한 뒤 그 블록을 렌더링하는 것이다.

### Key Components

- **`buildChatRenderBlocks(...)`**: `ChatMessage[]`를 순회하며 일반 메시지, grouped tool run, orphan tool result를 구분하는 순수 헬퍼
- **`RenderBlock` types**: `message`, `tool_run`, `orphan_tool_result` 세 가지 렌더 단위 정의
- **`ToolRunBlock`**: 한 번의 도구 실행 묶음을 요약 헤더 + step rows로 렌더링하는 새 컴포넌트
- **`ToolRunStepRow`**: 각 도구 호출의 compact row와 row-level detail disclosure 담당
- **`ToolResultDetail`**: 기존 `ToolCallBubble.tsx`의 상세 결과 렌더링을 분리해 재사용하는 stateless detail renderer
- **`ToolCallBubble` fallback**: 고아 결과, legacy 경로, 예외 상황에서 유지할 기존 렌더러

### Render Pipeline

```text
useMessageStore().messages
  -> buildChatRenderBlocks(messages, toolRunState, pendingToolCalls)
  -> RenderBlock[]
  -> ChatWindow renders blocks
     - message -> existing ChatMessage
     - tool_run -> ToolRunBlock
     - orphan_tool_result -> ToolCallBubble fallback
```

### Grouping Rules

1. 메시지를 순서대로 순회한다.
2. agent 메시지에 `tool_calls`가 있으면 새 `tool_run` block을 시작한다.
3. 해당 agent 메시지의 `content`가 비어 있지 않으면 `leadingContent`로 같은 block 상단에 유지한다.
4. 바로 뒤에 오는 연속된 `tool` 메시지만 forward scan 대상에 포함한다.
5. `tool_call_id`가 현재 run의 call ID와 일치할 때만 step result로 붙인다.
6. 첫 non-`tool` 메시지를 만나면 scan을 종료한다.
7. scan 중 매칭되지 않는 `tool` 메시지는 원래 순서를 유지한 `orphan_tool_result` block으로 남긴다.
8. 일부 step에 결과가 아직 없어도 같은 run 안에 incomplete step으로 남긴다.

### Active Run Detection

- transcript만으로 completed run grouping은 충분하다.
- live status는 `ChatWindow`에서 `toolRunState`와 `pendingToolCalls`를 함께 넘겨 마지막 `tool_run` block에만 적용한다.
- 1차 구현에서는 “가장 마지막 `tool_run` block만 active 후보”로 간주한다.
- 이후 auto-approved per-call in-flight 식별이 더 필요하면 `toolRunStore`에 transient current-call ids를 추가하는 것을 2차 개선으로 둔다.

### Status Mapping

| Condition | UI Status |
|-----------|-----------|
| 매칭된 tool result가 정상 출력 | `executed` |
| 결과 내용이 rejection/denial 성격 | `denied` |
| 결과 내용이 `Error:` 등 오류 성격 | `error` |
| 결과 없음 + active run + `pendingToolCalls` 포함 + `toolRunState`가 `tool_pending`/`tool_waiting` | `pending approval` |
| 결과 없음 + active run + `toolRunState === tool_running` | `running` |
| 결과 없음 + 과거 run 또는 기타 케이스 | `incomplete` |

denial/error 분류는 문자열 규칙을 중앙 helper에서 단일화하고 테스트로 고정한다.

### Preview Priority Rules

각 step row의 한 줄 요약은 아래 우선순위로 생성한다.

1. 오류/거부 텍스트
2. 구조화 가능한 결과 요약
   - 브라우저 결과: title, URL, snapshot element count
   - 파일 쓰기류: 대상 파일 또는 preview 가능 여부
3. 결과 텍스트 첫 줄
4. 결과가 없으면 args summary
5. 아무것도 없으면 tool name 기반 기본 문구

### Approval Semantics

- 1차 구현에서는 승인 의미를 바꾸지 않는다.
- 여러 waiting step이 같은 run에 보여도 승인 액션은 기존처럼 현재 pending set 전체에 대한 동작으로 유지한다.
- UI는 row-level로 강조하되, 행동 의미는 run-level/current pending batch와 동일해야 한다.

## Implementation Plan

### Step 1: Render Model Helper 도입
- **Actions**: `RenderBlock` 타입과 `buildChatRenderBlocks(...)` 순수 헬퍼를 추가하고, 일반 메시지/도구 실행 묶음/고아 결과 분류 로직을 구현한다.
- **Deliverables**: 새 helper 파일, 타입 정의, grouping/spec 주석
- **Dependencies**: 기존 `ChatMessage`, `toolRunStore` 상태 구조 이해

### Step 2: 상세 결과 렌더러 분리
- **Actions**: `ToolCallBubble.tsx`에서 브라우저 결과, `write_file` preview, raw result fallback 렌더링을 별도 `ToolResultDetail`로 추출한다.
- **Deliverables**: 재사용 가능한 stateless detail renderer
- **Dependencies**: Step 1과 독립 가능, 기존 detail UI parity 확인 필요

### Step 3: Compact Tool Run UI 구현
- **Actions**: `ToolRunBlock`과 compact step row 컴포넌트를 추가한다. 요약 헤더, step status chip, outcome-first preview, row detail disclosure를 구현한다.
- **Deliverables**: grouped tool run UI 컴포넌트 세트
- **Dependencies**: Step 1, Step 2

### Step 4: ChatWindow 렌더 경계 통합
- **Actions**: `ChatWindow.tsx`에서 raw `messages.map(...)` 대신 `RenderBlock[]`를 만들어 렌더한다. 일반 메시지는 기존 `ChatMessage`를 재사용하고, grouped run과 orphan fallback을 분기 처리한다.
- **Deliverables**: 새로운 render pipeline
- **Dependencies**: Step 1, Step 3

### Step 5: 스타일 및 상호작용 정리
- **Actions**: `App.css`에 dense grouped-run 스타일을 추가하고, block-level collapse, row-level disclosure, 모바일 truncation/wrapping을 튜닝한다.
- **Deliverables**: compact visual system, responsive adjustments
- **Dependencies**: Step 3, Step 4

### Step 6: 테스트 및 회귀 검증
- **Actions**: helper 단위 테스트와 grouped UI 컴포넌트 테스트를 추가한다. 기존 `ChatMessage` fallback 경로도 검증한다.
- **Deliverables**: grouping tests, rendering tests, approval parity tests
- **Dependencies**: Step 1-5

## Technical Considerations

- 원본 `messages`는 저장/전송 계약이므로 변형하지 않고, 화면용 block만 파생한다.
- grouping은 `ChatMessage` 내부가 아니라 `ChatWindow` 경계에서 수행해야 형제 메시지 기반 판단과 중복 제거가 가능하다.
- 결과 버블을 완전히 제거하지 않고 fallback 경로로 남겨야 malformed history에 안전하다.
- detail renderer는 상태 없는 하위 컴포넌트로 분리해 grouped UI와 legacy bubble이 함께 재사용하도록 만든다.
- collapse 상태는 grouped block 내부에서만 소유해 전역 상태 복잡도를 늘리지 않는다.

## Inspiration Credits

- **Tool Run Group**: 연속된 호출과 결과를 하나의 실행 컨테이너로 묶어 scan target을 줄인다.
- **Timeline Rows Instead of Full Cards**: 각 도구를 compact row로 바꾸고 카드 chrome 반복을 제거한다.
- **Result Attached to Originating Call**: 결과를 나중 별도 버블이 아니라 원래 호출 row 아래에 붙여 cause/effect를 복원한다.
- **Auto-Fold Successful Bursts**: 성공한 묶음은 기본 접힘으로 시작하고 문제 단계만 전면에 드러낸다.
