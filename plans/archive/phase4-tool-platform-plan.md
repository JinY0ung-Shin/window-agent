## Requirements Summary

### Problem Statement
현재 window-agent의 에이전트는 텍스트 응답만 가능하다. 파일 읽기/쓰기, 웹 검색, 시스템 명령 등 실제 작업을 수행할 수 없어 유용성이 제한된다. Phase 4는 OpenAI function-calling 기반 도구 시스템을 도입하여 에이전트를 "실행 가능한 어시스턴트"로 확장한다.

### Scope

**In Scope (6 items):**
1. Tool/Skill 시스템 — function calling 기반 도구 정의, 호출, 실행 루프
2. Allowlist/Approval 모델 — 도구별 권한 계층 (자동/승인 필요/차단)
3. 경량 운영/디버그 패널 — 도구 호출 로그 및 상태 모니터링
4. 백업/내보내기 — 대화 + 에이전트 설정 JSON/ZIP 내보내기
5. 메모리 노트 — 에이전트별 영속 노트 (DB v4)
6. 폴더 없는 에이전트 표시 개선 — dormant 상태 표시

**Out of Scope:**
- 동적 스크립트 래핑 (런타임 도구 생성)
- Agent-to-Agent 도구 위임
- MCP (Model Context Protocol) 서버 통합
- 외부 플러그인 마켓플레이스

## Architecture

### Approach: Bootstrap 패턴 확장 + 실행 상태 머신 + 권한 계층

기존 `bootstrapService.ts`의 function-calling 루프를 일반화하되, 스트리밍 환경에서의 **일시정지/재개(pause/resume) 상태 머신**을 도입하여 승인 흐름을 안전하게 처리한다.

### Key Components

- **ToolRegistry** (Frontend) — 에이전트별 도구 정의 로드 및 OpenAI tools 배열 생성
- **ToolExecutor** (Rust Backend) — 도구 이름에 따른 실제 실행 (파일 I/O, HTTP 등)
- **ToolRunStateMachine** (Frontend) — 도구 실행 흐름 상태 관리 (아래 상세)
- **PermissionGate** (Frontend) — 도구 권한 확인 및 인라인 승인 UI
- **ToolCallBubble** (Frontend) — 도구 호출/결과를 채팅에 표시하는 컴포넌트
- **DebugPanel** (Frontend) — 도구 호출 로그 사이드 패널
- **MemoryNoteStore** (Frontend + DB) — 에이전트별 메모리 노트 CRUD
- **ExportService** (Rust Backend) — 대화 + 에이전트 JSON/ZIP 내보내기/불러오기

### Tool Run State Machine (Critical Issue #3 해결)

도구 호출이 포함된 스트리밍 대화는 다음 상태를 가진다:

```
        ┌─────────────┐
        │   IDLE      │ ← 대화 시작 전 / 완료 후
        └──────┬──────┘
               │ sendMessage()
        ┌──────▼──────┐
        │  STREAMING  │ ← API 스트리밍 중 (텍스트 + tool_calls 청크 수신)
        └──────┬──────┘
               │ stream 완료 (tool_calls 감지)
        ┌──────▼──────┐
        │  TOOL_PENDING│ ← tool_calls 파싱 완료, 권한 확인 중
        └──────┬──────┘
               │ auto → 즉시 / confirm → 사용자 대기
        ┌──────▼──────┐
        │  TOOL_WAITING│ ← confirm tier 도구의 사용자 승인 대기
        └──────┬──────┘
               │ approve() / reject()
        ┌──────▼──────┐
        │  TOOL_RUNNING│ ← Rust 백엔드에서 도구 실행 중
        └──────┬──────┘
               │ 실행 완료
        ┌──────▼──────┐
        │  CONTINUING  │ ← 도구 결과로 다시 API 호출 → STREAMING으로 전이
        └─────────────┘
```

**핵심 규칙:**
- STREAMING → TOOL_PENDING 전이 시 현재 스트림 구독 정리 (done 이벤트 후)
- TOOL_WAITING 상태에서는 새 메시지 전송 차단 (ChatInput disabled)
- reject() 시 → 거부 메시지를 tool result로 추가 → CONTINUING → API가 대안 응답 생성
- 전체 루프 상한: max 10 iterations (STREAMING→CONTINUING 사이클 카운트)
- 비정상 종료 시 → IDLE로 강제 전이 + 에러 메시지

### Data Flow (도구 호출)

```
1. User sends message
2. chatStore.sendNormalMessage()
   → buildConversationContext() (기존, 도구 transcript 포함)
   → toolRegistry.getToolsForAgent(agentId) → OpenAI tools[] 배열 추가
3. State → STREAMING: Rust chat_completion_stream (tools 파라미터 포함)
4. 스트리밍 중 tool_calls 청크 누적 (delta.tool_calls[i].function.arguments 연결)
5. stream done 이벤트 수신 시:
   a. tool_calls가 없으면 → 일반 응답 저장 → IDLE
   b. tool_calls가 있으면 → State → TOOL_PENDING
6. TOOL_PENDING: 각 tool_call의 tier 확인
   a. 모든 auto → State → TOOL_RUNNING → 즉시 실행
   b. confirm 있으면 → State → TOOL_WAITING → 인라인 승인 UI
7. TOOL_RUNNING: Rust execute_tool 호출, 결과 수신
8. 결과를 messages에 추가 (assistant tool_calls msg + tool result msgs)
   → DB에 영속 저장 (role="assistant" with tool_calls / role="tool" with result)
9. State → CONTINUING → STREAMING (다시 API 호출)
10. tool_calls가 없는 최종 응답 수신 → 저장 → IDLE
11. 전체 도구 호출 이력을 tool_call_logs에 기록
```

## Tool Transcript Persistence (Critical Issue #1 해결)

### 메시지 저장 모델

**중요:** 현재 `messages.content`는 `NOT NULL` 제약이 있다. Assistant tool-call 행은 `content=""`(빈 문자열)을 사용한다.

도구 호출이 포함된 대화는 다음과 같이 messages 테이블에 저장된다:

```
# 일반 메시지 (기존)
{ role: "user",      content: "파일 읽어줘",         tool_call_id: NULL,       tool_name: NULL,        tool_input: NULL }
# Assistant가 도구 호출 (content="" 빈 문자열, NOT NULL 유지)
{ role: "assistant", content: "",                     tool_call_id: NULL,       tool_name: NULL,        tool_input: '[{"id":"call_abc","type":"function","function":{"name":"read_file","arguments":"{\"path\":\"/tmp/test.txt\"}"}}]' }
# 도구 실행 결과
{ role: "tool",      content: "파일 내용입니다...",   tool_call_id: "call_abc", tool_name: "read_file", tool_input: NULL }
# 최종 어시스턴트 응답
{ role: "assistant", content: "파일을 읽었습니다...", tool_call_id: NULL,       tool_name: NULL,        tool_input: NULL }
```

### End-to-End 메시지 Contract

도구 메시지가 지나가는 모든 레이어의 데이터 형태를 명시한다:

**1. DB Layer (Rust models.rs)**
```rust
pub struct Message {
    pub id: String,
    pub conversation_id: String,
    pub role: String,           // "user" | "assistant" | "tool"
    pub content: String,        // NOT NULL, 빈 문자열 허용
    pub tool_call_id: Option<String>,  // role="tool"일 때 설정
    pub tool_name: Option<String>,     // role="tool"일 때 설정
    pub tool_input: Option<String>,    // role="assistant"일 때 tool_calls JSON
    pub created_at: String,
}
```

**2. Rust → Frontend (Tauri IPC)**
```typescript
interface MessageFromDB {
  id: string;
  conversation_id: string;
  role: "user" | "assistant" | "tool";
  content: string;
  tool_call_id: string | null;
  tool_name: string | null;
  tool_input: string | null;  // JSON string of ToolCall[]
  created_at: string;
}
```

**3. Frontend UI Model (types.ts)**
```typescript
interface ChatMessage {
  id: string;
  type: "user" | "agent" | "tool";
  content: string;
  tool_calls?: ToolCall[];     // assistant 메시지: 파싱된 도구 호출
  tool_call_id?: string;       // tool 메시지: 어떤 호출의 결과인지
  tool_name?: string;          // tool 메시지: 도구 이름
  status: "pending" | "complete" | "error";
}
```

**4. Frontend → OpenAI API (요청 메시지 배열)**
```typescript
// Assistant with tool_calls
{ role: "assistant", content: "" || null, tool_calls: [{ id, type: "function", function: { name, arguments } }] }
// Tool result
{ role: "tool", tool_call_id: "call_abc", content: "결과 텍스트" }
// 일반 메시지 (기존)
{ role: "user" | "assistant", content: "텍스트" }
```

**5. 변환 흐름:**
- DB → Frontend UI: `tool_input` JSON 파싱 → `tool_calls` 배열. `role="tool"` → `type="tool"`
- Frontend UI → API: `tool_calls` 배열 → OpenAI format. `type="tool"` → `role="tool"`
- API Response → DB: `tool_calls` → JSON 직렬화 → `tool_input`. `content` 없으면 `""`(빈 문자열)

**6. 영향받는 함수/모듈 (변경 필요):**

| Layer | File | Function/Struct | 변경 내용 |
|-------|------|----------------|-----------|
| DB Write | operations.rs | `save_message()` | tool_call_id, tool_name, tool_input 파라미터 추가 |
| DB Read | operations.rs | `get_messages()` | 확장 컬럼 SELECT, Message 구조체 매핑 |
| Rust Model | models.rs | `Message` | 3개 Optional 필드 추가 |
| Tauri CMD | chat_commands.rs | `save_message`, `get_messages` | 확장 필드 전달 |
| IPC Bridge | tauriCommands.ts | `saveMessage()`, `getMessages()` | 확장 필드 타입 추가 |
| Types | types.ts | `ChatMessage`, `MessageRole` | tool 관련 필드 + "tool" role 추가 |
| Chat Helpers | chatHelpers.ts | `buildChatMessages()` | tool 메시지 → OpenAI 형식 변환 |
| Chat Store | chatStore.ts | `sendNormalMessage()`, 로드 로직 | tool_input 파싱/직렬화, tool 메시지 처리 |

### 재로드 및 API 재전송

대화 재로드 시:
1. DB에서 messages를 순서대로 로드 (tool_call_id, tool_name, tool_input 포함)
2. `role="assistant"` + `tool_input IS NOT NULL` → JSON.parse → `tool_calls` 배열로 복원
3. `role="tool"` → `type="tool"` + `tool_call_id` + `tool_name` 매핑
4. `buildConversationContext()`가 이 메시지들을 토큰 예산 내에서 OpenAI 형식으로 포함
5. 파싱 실패 시 graceful degradation: `tool_input`을 `content`로 표시 (텍스트 fallback)

### Rolling Summary와의 통합

- 도구 호출 메시지도 토큰 계산에 포함
- Summary 생성 시 도구 호출/결과를 자연어로 요약 (예: "read_file로 /tmp/config.json을 읽음 → 설정 값 확인")
- Summary가 커버하는 범위의 도구 메시지는 컨텍스트에서 제외 (기존 로직과 동일)
- 도구 메시지 토큰 추정: tool_input JSON + tool result content 모두 포함

## TOOLS.md Contract (Critical Issue #2 해결)

### 파일 형식

```markdown
# Tools

## read_file
- description: 지정 경로의 파일 내용을 읽습니다
- tier: auto
- parameters:
  - path (string, required): 파일 경로

## write_file
- description: 지정 경로에 파일을 씁니다
- tier: confirm
- parameters:
  - path (string, required): 파일 경로
  - content (string, required): 파일 내용

## list_directory
- description: 디렉토리 내 파일 목록을 조회합니다
- tier: auto
- parameters:
  - path (string, required): 디렉토리 경로

## web_search
- description: 웹에서 정보를 검색합니다
- tier: confirm
- parameters:
  - query (string, required): 검색 쿼리

## memory_note
- description: 에이전트 메모리 노트를 관리합니다
- tier: auto
- parameters:
  - action (string, required): create | read | update | delete
  - title (string, required): 노트 제목
  - content (string, optional): 노트 내용
```

### 파싱 규칙
- `## tool_name` 헤더로 도구 구분
- `- description:` → OpenAI function description
- `- tier:` → auto | confirm | deny (없으면 기본 confirm)
- `- parameters:` → OpenAI function parameters 스키마로 변환
  - `name (type, required/optional): description` 형식
- 알 수 없는 도구 이름 → 무시 (경고 로그)
- 내장 도구 이름과 일치해야 실행 가능 (임의 이름 불가)

### 백엔드 파일 접근 변경 (Enabling Change)

현재 `agent_commands.rs`의 `read_persona_file` / `write_persona_file`은 4개 파일명만 허용:
```rust
const ALLOWED_FILES: [&str; 4] = ["IDENTITY.md", "SOUL.md", "USER.md", "AGENTS.md"];
```

**변경:** `TOOLS.md`를 5번째 허용 파일로 추가:
```rust
const ALLOWED_FILES: [&str; 5] = ["IDENTITY.md", "SOUL.md", "USER.md", "AGENTS.md", "TOOLS.md"];
```

### TOOLS.md 없는 에이전트 동작
- `toolRegistry.getToolsForAgent()` → 빈 배열 `[]` 반환
- API 호출 시 `tools` 파라미터 생략 → 기존 텍스트 전용 동작
- 부트스트랩 시 기본 TOOLS.md 템플릿 자동 생성 (5개 내장 도구 모두 포함)

    { "id": "...", "role": "user", "content": "...", "tool_call_id": null, "tool_name": null, "tool_input": null, "created_at": "..." },
    { "id": "...", "role": "tool", "content": "...", "tool_call_id": "call_abc", "tool_name": "read_file", "tool_input": null, "created_at": "..." }

**에이전트 내보내기 (ZIP):**

agent-export-{name}.zip

### Import 충돌 해결 규칙

| 충돌 유형 | 감지 방법 | 해결 전략 |
|-----------|-----------|-----------|
| Agent ID 중복 | DB에서 id로 조회 | 새 UUID 발급, folder_name에 `-imported` 접미사 |
| Agent folder_name 중복 | 파일시스템 + DB 조회 | folder_name에 `-imported-{N}` 접미사 (N=1부터 증가) |
| Conversation ID 중복 | DB에서 id로 조회 | 새 UUID 발급, agent_id를 import된 새 agent_id로 매핑 |
| Message ID 중복 | DB에서 id로 조회 | 새 UUID 발급 |
| Memory Note ID 중복 | DB에서 id로 조회 | 새 UUID 발급, agent_id를 새 agent_id로 매핑 |

**ID Remapping 규칙:**
1. Agent 먼저 import → old_agent_id → new_agent_id 매핑 생성
2. Conversation import 시 agent_id를 매핑으로 치환
3. Message import 시 conversation_id를 매핑으로 치환
4. Memory Note import 시 agent_id를 매핑으로 치환
5. 모든 매핑은 import 트랜잭션 내에서 처리 (실패 시 전체 롤백)

**tool_call_logs Export 정책:**
- tool_call_logs는 내보내기에서 **제외** (디버그 용도이며 재현 불필요)
- 도구 호출 이력은 messages 테이블의 transcript로 충분히 재현 가능
- Import 후 도구가 다시 실행되면 새 로그가 생성됨

**Import 결과 리포트:**
```json
{
  "agents_imported": 1,
  "conversations_imported": 3,
  "messages_imported": 45,
  "memory_notes_imported": 5,
  "id_remappings": { "agents": 1, "conversations": 2 },
  "warnings": ["folder_name 'assistant' already existed, renamed to 'assistant-imported-1'"]
}
```

## DB Schema Changes (Migration v4)

```sql
-- 메시지 테이블 확장: 도구 호출 지원
ALTER TABLE messages ADD COLUMN tool_call_id TEXT;
ALTER TABLE messages ADD COLUMN tool_name TEXT;
ALTER TABLE messages ADD COLUMN tool_input TEXT;

-- 메모리 노트 테이블
CREATE TABLE memory_notes (
  id TEXT PRIMARY KEY,
  agent_id TEXT NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_memory_notes_agent_id ON memory_notes(agent_id);

-- 도구 호출 로그 테이블
CREATE TABLE tool_call_logs (
  id TEXT PRIMARY KEY,
  conversation_id TEXT NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
  message_id TEXT REFERENCES messages(id) ON DELETE SET NULL,
  tool_name TEXT NOT NULL,
  tool_input TEXT NOT NULL,
  tool_output TEXT,
  status TEXT NOT NULL DEFAULT 'pending',
  duration_ms INTEGER,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_tool_call_logs_conversation ON tool_call_logs(conversation_id);
```

### Migration 검증 및 롤백

- **사전 검증:** 마이그레이션 전 `PRAGMA table_info(messages)`로 기존 스키마 확인, 이미 컬럼 있으면 스킵
- **원자성:** 모든 DDL을 단일 트랜잭션으로 실행 (SQLite는 DDL 트랜잭션 지원)
- **부분 실패:** 트랜잭션 실패 시 전체 롤백, 앱 시작은 v3 스키마로 계속 가능
- **기존 데이터:** ALTER TABLE ADD COLUMN은 기존 행에 NULL 할당 → 기존 메시지 무영향
- **다운그레이드:** 지원하지 않음. v4→v3 롤백 필요 시 수동 백업 복원 (내보내기 기능 활용)
- **마이그레이션 실패 시 런타임 동작:** v4 마이그레이션 실패 시 앱은 v3 스키마로 계속 시작. 이 경우:
  - 도구 호출 기능: **비활성** (toolRegistry가 빈 배열 반환, 도구 관련 UI 숨김)
  - 메모리 노트: **비활성** (MemoryBar 숨김)
  - 디버그 패널: **비활성** (버튼 숨김)
  - 내보내기: **기존 데이터만** 가능 (도구/메모리 필드 제외)
  - 기존 채팅 기능: **정상 동작** (Phase 1-3 기능 무영향)
  - 앱 설정에 "Phase 4 기능 비활성 (DB 업그레이드 필요)" 경고 표시

## Technical Considerations

- **토큰 예산**: 도구 정의 5개 ≈ 300-500 토큰 + 메모리 노트 최대 500 토큰 → `MAX_CONTEXT_TOKENS`에서 차감. 총 오버헤드 최대 ~1000 토큰 (8000 중 12.5%)
- **스트리밍 + 도구 호출 혼합**: delta.tool_calls[i]를 index 기반으로 누적. arguments는 문자열 청크 연결. 기존 rAF 배치 로직과 통합
- **도구 실행 타임아웃**: 각 도구 30초 (tokio::time::timeout). 전체 루프 max 10 iterations
- **Backward Compatibility**: TOOLS.md 없으면 tools=[] → API에 tools 파라미터 생략 → 기존 동작
- **보안**: allowed_roots prefix 검사 + symlink resolve. 셸 실행은 Phase 4 스코프 외
- **Summary 통합**: 도구 메시지도 요약 범위에 포함. 요약 텍스트에 도구 사용 이력 반영
